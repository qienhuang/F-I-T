# FIT 版本策略（Versioning Policy）

本文件定义 FIT 框架如何随时间演化，目的是在“稳定性/范围/兼容性”上给读者、评审与贡献者一个可审计的约定，避免语义漂移。

---

## 1. 核心原则

FIT 采用 **closed-core, open-edge** 的演化模型：

- **核心（core）** 刻意保守：只有在明确压力下才修改。
- **边缘（edges）** 预期增长：证据、应用、领域案例会持续扩展。

---

## 2. 版本序列语义

### 2.x：核心稳定序列

2.x 代表一个相对稳定的理论核心。

特征：
- 不新增核心变量（仍是 Force / Information / Time / Constraint / Phase）
- 不重定义 MCC（Minimal Coherent Core）
- 不做“无声明的语义改动”

允许的变化：
- 澄清（clarifications）
- 由明确反例驱动的精炼（counterexample-driven refinements）
- estimator-conditional 的条件化重述
- 边界条件补齐与防误用说明

2.x 中任何更改都必须：
- 被文档化
- 被版本化
- 链接到触发该更改的证据、失败模式或推理压力

---

### 3.x：形式化扩展序列

3.x 预留给真正的形式化扩展，例如：
- 连续时间版本
- 随机微分方程（SDE）版本
- 2.x 无法表达的新数学结构

从 2.x 升级到 3.x 表示：
- 形式层级发生改变
- 而不仅是增量改进

---

## 3. 什么不触发核心版本升级

以下内容通常不需要提升 core 版本：
- 新案例/新实验
- 新的证据材料
- 新应用领域
- 在 Core Card 语义范围内的新解释

这些属于 edge 扩展。

---

## 4. 向后兼容承诺

- 核心文档（Core Card / MCC / Phase Algebra）在同一 major 版本序列内尽量保持向后兼容。
- 若必须弃用某些表述，应明确标注并保留可追溯性（不要悄悄删改）。

---

## 5. 修订纪律

每一次核心修订都应包含：
- 改了什么（what changed）
- 为什么必须改（why necessary）
- 哪些证据/推理压力迫使更改（forcing evidence/reasoning）

仅仅出于叙事偏好（narrative preference）不足以触发核心改动。

---

## 6. 总结

- 2.x 天生保守、以稳定为目标
- 更改应由反例驱动，而不是由“新点子”驱动
- 读者可以在不担心语义漂移的前提下引用 2.x

该策略是 FIT 自我治理与长期可信度的一部分。
